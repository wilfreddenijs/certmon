# Toolbelt auto-upload UI with device progress and cancellation

## Objective

Add a desktop Toolbelt batch-upload workflow that turns the existing `devices.txt` Local CA device/certificate mapping into a visible, selectable UI list; automatically performs safe dry-runs; allows explicit upload only for eligible selected devices; shows per-device progress; supports stop-after-current-device cancellation; and persists latest per-device results.

## Context Required

Before implementing, read:

- `.planning/phases/certmon-03-toolbelt-auto-upload-ui-with-device-progress-and-cancellatio/03-SPEC.md`
- `.planning/phases/certmon-03-toolbelt-auto-upload-ui-with-device-progress-and-cancellatio/03-CONTEXT.md`
- `toolbelt_uploader.py`
- `app.py`
- `templates/index.html`
- `certmon/artifacts.py`
- `certmon/vault.py`
- `certmon/db.py`

## Non-Goals

- No server-mode/multi-user role work.
- No non-Extron automatic upload adapter.
- No custom replacement for Extron Toolbelt.
- No force-abort during an active device operation.
- No full historical audit log; persist latest per-device status only.

## Tasks

### Task 1: Add a machine-readable Toolbelt CLI contract

Update `toolbelt_uploader.py` without removing existing standalone behavior.

- [ ] Add `--jsonl` mode that emits one JSON object per line to stdout.
- [ ] Emit run-level events: `run_started`, `run_finished`, `run_failed`.
- [ ] Emit device-level events: `device_pending`, `device_started`, `dry_run_ok`, `dry_run_failed`, `upload_ok`, `upload_failed`, `credentials_needed`, `serial_column_missing`, `device_skipped`, `device_cancelled`.
- [ ] Add `--stop-file <path>` support; check between devices and stop before starting the next one.
- [ ] Add a credential input mechanism that does not expose secrets on the command line, such as a temporary credential JSON file with restricted lifetime.
- [ ] Keep dry-run as default and require `--commit` for real upload.
- [ ] Preserve current human log behavior for troubleshooting.

Verification:

```powershell
pytest tests/test_toolbelt_uploader.py -v
```

Acceptance criteria:

- [ ] Existing CLI examples still work.
- [ ] Dry-run JSONL contains no `--commit` behavior.
- [ ] Stop-file prevents the next device from starting.
- [ ] JSONL output never includes passwords or private key contents.

### Task 2: Add backend Toolbelt batch service and routes

Create a small backend orchestration layer, preferably `certmon/toolbelt.py`, and connect it through `app.py`.

- [ ] Build Toolbelt target rows from Local CA certificate metadata, matching the existing `/api/ca/devices-txt` mapping.
- [ ] Include device selector, certificate ID, display label, Local CA/Extron certificate status, latest dry-run status, latest upload status, and selected default.
- [ ] Materialize Extron `combined.pem` private artifact server-side into a temporary run directory for Toolbelt only.
- [ ] Delete temporary PEM and credential files on success, failure, and cancellation cleanup.
- [ ] Add routes:
  - `GET /api/toolbelt/devices`
  - `POST /api/toolbelt/dry-run`
  - `POST /api/toolbelt/upload`
  - `GET /api/toolbelt/runs/<run_id>`
  - `POST /api/toolbelt/runs/<run_id>/stop`
  - `PATCH /api/toolbelt/devices/<selector>/credentials`
- [ ] Refuse upload for devices whose latest dry-run status is not OK.
- [ ] Handle Toolbelt missing/elevation mismatch as clear user-facing errors.
- [ ] Ensure no route accepts private PEM/key material in request bodies.

Verification:

```powershell
pytest tests/test_toolbelt_api.py tests/test_ca_api.py tests/test_deployment.py -v
```

Acceptance criteria:

- [ ] `/api/toolbelt/devices` matches `/api/ca/devices-txt` mappings.
- [ ] Dry-run starts a subprocess without `--commit`.
- [ ] Upload starts a subprocess with `--commit` only for selected dry-run-OK devices.
- [ ] Stop route creates/uses the safe stop signal.
- [ ] Temporary PEM and credential files are removed after run cleanup.

### Task 3: Persist latest per-device status and encrypted credentials

Extend persistence with a minimal, testable model.

- [ ] Store latest dry-run and upload result per device selector/certificate mapping: status, timestamp, message, run ID, and mode.
- [ ] Store per-device credentials encrypted through existing Vault/Database secret patterns.
- [ ] Use fallback order:
  1. saved encrypted per-device credentials;
  2. default `admin` / `extron`;
  3. `admin` / serial number when Toolbelt serial is available;
  4. UI prompt for custom credentials.
- [ ] Do not persist serial number as a password unless the user explicitly saves credentials for that device.
- [ ] Do not include credentials, serial-number-as-password attempts, or PEM paths in UI logs or persistent status messages.


Acceptance criteria:

- [ ] Latest status survives app restart.
- [ ] Passwords are not returned by any API response.
- [ ] Saved credentials are encrypted and can be used by later dry-run/upload.
- [ ] Status messages are useful but sanitized.

- [ ] Real upload requires explicit click and processes only checked eligible devices.
- [ ] Stop button prevents next device from starting after current device completes.
- [ ] Latest per-device status persists after restart.
- [ ] Per-device credentials are encrypted and never returned to browser responses.

- [ ] Missing Toolbelt and elevation mismatch are shown as clear UI errors.
